# Writing GitLab Agent Briefs

Read this file before posting a brief on an issue or merge request that is moving to `ready-for-agent` or `ready-for-human`.

An agent brief is the authoritative implementation contract. The original description and discussion provide history; the most recent approved brief defines the work to perform.

For an issue, describe the behavior to build or repair. For an MR, describe what remains to finish, correct, or verify in the existing diff.

## Principles

### Prefer durability over location-specific precision

Write a brief that remains useful if files move or internals are refactored.

- Describe public interfaces, domain types, configuration shapes, and behavioral contracts.
- Name stable types or function signatures when they are part of the contract.
- Do not cite file paths or line numbers.
- Do not assume the current internal structure must be preserved.

### Specify behavior, not an implementation recipe

Describe observable outcomes and constraints. Let the implementing agent explore the current codebase and choose the smallest appropriate design.

Good: “When a signed-in user requests an archived invoice, the API returns the documented not-found response.”

Bad: “Add an `if` statement to the invoice controller.”

### Make completion independently verifiable

Every acceptance criterion must be concrete and testable. Include success, error, and compatibility behavior when relevant.

### State boundaries explicitly

List adjacent changes that are not part of this work. This prevents accidental expansion of scope.

### Separate evidence from inference

State whether the claim was confirmed, not reproduced, or could not be verified. Never turn a suspected root cause into a contractual fact without evidence.

## Agent-ready template

```markdown
> *This was generated by AI during triage.*

## Agent Brief

**Category:** bug | enhancement

**Summary:** One sentence describing the required outcome.

**Verification:** Confirmed | Not reproduced | Insufficient detail

Briefly state the evidence and conditions.

**Current behavior:**

Describe the observable status quo. For an MR, describe what the existing diff already accomplishes and what remains incomplete.

**Desired behavior:**

Describe the result after the work is complete, including important edge cases and error behavior.

**Key interfaces and contracts:**

- Stable interface, type, command, API, or configuration contract and the required behavior

**Acceptance criteria:**

- [ ] Independently testable outcome
- [ ] Relevant error or edge case
- [ ] Existing supported behavior remains unchanged

**Out of scope:**

- Adjacent behavior that must not be changed
```

## Human-ready addition

For `ready-for-human`, use the same template and add:

```markdown
**Why human handling is required:**

Explain the judgment call, external access, design decision, manual verification, or merge responsibility that prevents autonomous completion.
```

## Merge request guidance

For an MR:

- Treat the submitted diff as the starting point.
- Identify which claimed behaviors were verified.
- Describe gaps in terms of outcomes, not requested edits.
- Preserve the contributor's established design unless it violates a documented contract.
- Include acceptance criteria for regressions and compatibility.

## Reject weak briefs

Do not post a brief that lacks any of these:

- a clear category and outcome
- current and desired behavior
- verification status
- stable interfaces or contracts when relevant
- testable acceptance criteria
- explicit scope boundaries
