# Triage GitLab Issues and Merge Requests

Move work on the current GitLab project through a small, explicit state machine. Treat an external merge request as an issue with attached code: use the same categories and states, but verify its existing diff instead of assuming implementation starts from scratch.

Do not create a new issue merely because the user reports a problem. This workflow operates on existing GitLab issues and merge requests unless the user explicitly asks to file new work.

Start every comment or issue description posted during triage with:

> *This was generated by AI during triage.*

## Supporting guidance

- Read [AGENT-BRIEF.md](AGENT-BRIEF.md) before posting a `ready-for-agent` or `ready-for-human` brief.
- Read [OUT-OF-SCOPE.md](OUT-OF-SCOPE.md) before evaluating or recording a rejected enhancement.

## GitLab setup

Use the repository's configured `glab` command or wrapper. Repository instructions take precedence; for example, a self-managed GitLab project may provide a wrapper transparently under the `glab` command.

Before reading or changing tracker state:

```sh
glab auth status
glab repo view
```

Use `--repo <group/project>` whenever the current directory is not the intended project. In GitLab notation, `#42` means issue 42 and `!42` means merge request 42. If the user supplies only `42`, resolve the type from their wording or ask rather than guessing.

Read repository instructions and list the project's labels before applying state:

```sh
glab label list --output json
```

Use the canonical roles below when those labels exist. If repository documentation maps them to different labels, use that mapping. If required labels or a mapping are absent, report the intended role and ask before creating or substituting labels.

## Roles and invariants

Assign exactly one category role:

- `bug` — existing behavior is broken
- `enhancement` — a new capability or improvement

Assign exactly one state role:

- `needs-triage` — awaiting maintainer evaluation
- `needs-info` — blocked on specific information from the reporter
- `ready-for-agent` — fully specified and suitable for autonomous implementation
- `ready-for-human` — ready for human work or review
- `wontfix` — will not be actioned

For a merge request, `ready-for-agent` means an agent brief describes what remains to be changed in the existing diff. `ready-for-human` means the MR is ready for human handling, such as judgment, manual verification, or merge review.

If an item has multiple category or state roles, stop and show the conflict before changing anything. Let the maintainer choose the intended role.

Normal state transitions are:

```text
untriaged -> needs-triage -> needs-info | ready-for-agent | ready-for-human | wontfix
needs-info -> needs-triage when the reporter responds
```

The maintainer may override the state at any time. Flag an unusual transition and confirm the intended mutation before applying it.

## Find work needing attention

When asked what needs attention, query GitLab and present these buckets oldest first:

1. Open issues with no triage state — never triaged.
2. Open issues labeled `needs-triage` — awaiting evaluation.
3. Open issues labeled `needs-info` where the reporter has commented since the most recent triage note — ready for re-evaluation.

Use `glab issue list` with JSON output, label filters, ascending creation order, and enough pagination for the project. Inspect comments for `needs-info` candidates rather than treating every recently updated item as a reporter response.

Include merge requests in bulk discovery only when repository instructions define which MRs belong in triage, such as MRs from external contributors. Tag each result `[issue]` or `[MR]`. An explicitly named MR is always in scope regardless of its author.

Show the count and a one-line summary for each bucket, then let the maintainer choose an item. Do not modify labels during discovery.

## Triage a specific item

### 1. Gather the complete context

Read the body, comments, labels, author, dates, and system activity:

```sh
glab issue view <iid> --comments --system-logs --output json
glab mr view <iid> --comments --system-logs --output json
```

For an MR, also inspect its diff, pipeline status, review discussions, commits, and target branch. Preserve any local user changes if checking out the MR; do not force checkout over a dirty worktree.

Parse previous AI triage notes and maintainer decisions. Do not repeat questions already answered.

Explore the codebase using its domain terminology and respect relevant ADRs and repository instructions. Perform all of these checks:

- **Existing implementation:** Search by domain concept, not only the reporter's wording. Report where and how you searched. If the requested behavior already exists, recommend `wontfix` as already implemented.
- **Duplicate tracker work:** Search open and closed GitLab issues and MRs for the same underlying request. Surface likely duplicates to the maintainer.
- **Prior rejection:** Read `.out-of-scope/*.md` and surface conceptually similar decisions.

### 2. Recommend without mutating

Present:

- the recommended category and state
- the evidence and reasoning
- a concise codebase summary
- whether the behavior already exists, duplicates prior work, or matches a prior rejection
- what remains uncertain

Wait for maintainer direction before changing labels, commenting, editing repository files, or closing anything.

### 3. Verify the claim

Verify before asking the reporter to elaborate:

- For a bug, follow the reproduction steps and run the smallest relevant tests or commands.
- For an enhancement, confirm the capability is absent and identify the observable contract it would introduce.
- For an MR, check out or otherwise inspect the proposed diff, then run the relevant tests and confirm that the diff does what its description claims.

Report one verification status:

- **Confirmed** — reproduced or directly demonstrated, with the observable behavior and relevant code path
- **Not reproduced** — attempted under stated conditions but did not observe the claim
- **Insufficient detail** — verification cannot be attempted reliably without specific missing information

Do not present static code inference as successful reproduction.

### 4. Clarify only when needed

If important product or domain decisions remain, ask one focused question at a time. Capture each answer so the next question incorporates prior decisions. Keep terminology consistent with the project's domain model and update repository context or ADRs only when the maintainer explicitly approves those edits.

Questions must be specific and actionable. Prefer “Which of these responses should the API return when the record is archived?” over “Can you provide more information?”

### 5. Apply the approved outcome

Before acting, summarize the exact mutations: labels to add/remove, comment to post, repository file to edit, and whether the item will be closed. Then apply only the approved outcome.

Use the appropriate GitLab commands:

```sh
glab issue update <iid> --label <label> --unlabel <old-label>
glab mr update <iid> --label <label> --unlabel <old-label>
glab issue note <iid> --message "<comment>"
glab mr note create <iid> --message "<comment>" --resolvable=false
glab issue close <iid>
glab mr close <iid>
```

When passing multiline Markdown, use actual newline characters, not literal `\n` sequences. Prefer a quoted multiline shell value or stdin where supported. Re-read the item after mutation and verify that it has exactly one category and one state label.

Apply each state as follows:

- **`ready-for-agent`:** Post an agent brief using [AGENT-BRIEF.md](AGENT-BRIEF.md), then apply the category and state labels.
- **`ready-for-human`:** Post the same brief structure, explicitly stating why human handling is required, then apply the labels.
- **`needs-info`:** Post the triage-notes template below, mention the reporter, and apply the labels.
- **`wontfix`, already implemented:** Point to the existing behavior, apply the labels, and close. Do not add an out-of-scope record.
- **`wontfix`, rejected bug:** Explain the decision politely, apply the labels, and close.
- **`wontfix`, rejected enhancement:** Follow [OUT-OF-SCOPE.md](OUT-OF-SCOPE.md), link the resulting record from the comment, apply the labels, and close.
- **`needs-triage`:** Apply the labels. Add a comment only when partial findings would otherwise be lost.

## Direct state overrides

If the maintainer explicitly asks to move an item to a state, trust the instruction. Confirm the concrete mutations and act without running a full clarification interview.

When moving an item to `ready-for-agent` without an existing brief, ask whether the maintainer wants a brief posted. Do not silently label underspecified work as agent-ready.

## Needs-info comment

```markdown
> *This was generated by AI during triage.*

## Triage Notes

**What we have established:**

- Confirmed fact or decision

**What we still need from @reporter:**

- Specific, actionable question
```

Record everything already established so future sessions do not lose context.

## Resume prior triage

When prior triage notes exist, compare their timestamp with later reporter activity, determine which questions were answered, and present the updated situation before continuing. Never re-ask a resolved question.
